# サービス戦略と運用方法

各サービスをどう活用していくか、類似サービスをどう使い分けていくか、その戦略や運用方法をまとめます。

---

## 0. 基本方針（迷ったらここだけ）

### ハブとスポークの考え方

- **ハブ（唯一の入口）**：`fluxrozin.com`
- **正本（成果物の正）**：`GitHub`
- **制作を支える道具**：`Google`（Gmail / YouTube / Gemini / NotebookLM）
- **蓄積（脳内整理）**：`Notion`
- **長文の資産化**：`Zenn / Qiita`
- **速報の配信**：`X`（補助で Bluesky / Threads）
- **守り**：`Cloudflare` + `Microsoft(復旧窓口)` + `Security Baseline`

**「1 つ作ったら、5 回使う（再利用）」が断続運用のコツです。**

---

## 1. 全体戦略：中心 → 横展開の考え方

### 1-1. "中心に据える"とは

断続的に続けるには、毎回触る場所を減らします。中心は次の 2 つだけに固定します。

1. **入口の中心**：`fluxrozin.com`
2. **成果物の中心**：`GitHub`（コード/ノート/README/Issue/Release）

ここに「戻れる」状態を作ると、SNS やプラットフォームの流行に左右されにくくなります。

### 1-2. "横展開"とは

中心で作ったものを、各サービスの特性に合わせて「薄く切って配信」します。

- **長文**：Zenn/Qiita（検索に残る）
- **スライド**：Docswell/Speaker Deck（登壇・共有に強い）
- **動画**：YouTube / TikTok（デモ・理解に強い）
- **短文**：X / Bluesky / Threads（拡散・出会い）
- **AI 成果物**：Hugging Face / Civitai（作品として残る）
- **実験記録**：W&B / Kaggle（研究ログとして残る）
- **Q&A**：Stack Overflow / Teratail（困ってる人を救う＝長期資産）
- **キャリア**：LAPRAS / LinkedIn / Wantedly（成果の束ね先）

---

## 2. Google 中心で運用して良いか？

### 2-1. Google に寄せると楽になる領域

- **調査・要約**：NotebookLM（資料を入れて"自分用の教科書"にする）
- **構成・叩き台**：Gemini（アウトライン/見出し/チェックリスト生成）
- **連絡と本人性**：Gmail（各サービスの登録メール）
- **映像の母艦**：YouTube（デモ動画・講義・制作ログ）

### 2-2. Google に寄せすぎない領域（分散して守る）

- **入口**：`fluxrozin.com`（Google アカ停止でも残る）
- **成果物**：`GitHub`（Markdown/コードは持ち運べる）
- **復旧窓口**：`Microsoft`（Google が死んでも戻せる）

**「Google は作業机、ドメインと GitHub は金庫」くらいに考えると安定します。**

---

## 3. 小規模・断続運用の"最小セット"

### 3-1. 常に触る（Tier 0）

- **`fluxrozin.com`**：入口・作品一覧（更新は月 1 でも OK）
- **`GitHub`**：成果物の正（README だけでも価値がある）
- **`Google`**：制作（Gemini/NotebookLM）と YouTube
- **`Notion`**：ネタ帳・下書き置き場
- **`Zenn / Qiita`**：長文の置き場（どちらかを主戦場に）

### 3-2. 出す時だけ触る（Tier 1）

- **X / Bluesky / Threads**：告知・断片
- **Speaker Deck / Docswell**：スライドがある時
- **YouTube / TikTok**：動画がある時
- **Hugging Face / Civitai**：AI 成果物がある時
- **Docker Hub / PyPI / Postman**：配布物がある時

### 3-3. 余裕がある時だけ（Tier 2）

- **Kaggle / W&B**：研究・実験をやる時
- **Stack Overflow / Teratail**：回答できる時
- **connpass**：参加/登壇した時
- **LAPRAS / LinkedIn / Wantedly**：月 1 で棚卸し
- **AtCoder**：筋トレ

## 4. サービス別の使い方（勝ち筋 / やらないこと / 頻度）

### 4-1. ドメイン・ID

#### fluxrozin.com（本城）
- **勝ち筋**：全リンクの回収、代表作の展示、連絡先
- **やらない**：毎日更新、長文ブログの全部を自前で抱える
- **頻度**：月 1 更新 + 新作が出た時

#### Cloudflare（城壁）
- **勝ち筋**：DNS/セキュリティ、Workers/Pages で軽い公開
- **やらない**：複雑な運用（最初は最小構成）
- **頻度**：初期設定後は放置、四半期で見直し

#### Google（主要アカウント）
- **勝ち筋**：Gmail/YouTube、Gemini/NotebookLM で制作効率
- **やらない**：成果物の唯一保管場所にする
- **頻度**：毎回使う

#### Microsoft（バックアップ窓口）
- **勝ち筋**：復旧用メール、予備連絡先
- **やらない**：主要活動の中心にする
- **頻度**：月 1 で受信確認

#### Gravatar（顔）
- **勝ち筋**：アイコンの統一
- **やらない**：サービスごとにバラバラな顔
- **頻度**：年 1 or 変更時

### 4-2. 開発・ツール

#### GitHub（心臓部）
- **勝ち筋**：README、Release、Issue で"残る"
- **やらない**：非公開で抱え込む、成果物が散る
- **頻度**：毎回

#### Docker Hub（コンテナ）
- **勝ち筋**：動くものを 1 コマンドで配る
- **やらない**：全部コンテナ化（必要な時だけ）
- **頻度**：配布が必要な時

#### PyPI（ライブラリ）
- **勝ち筋**：小さな便利関数でも公開して検索に残す
- **やらない**：完璧にしてから出す病
- **頻度**：まとまった時

#### Postman（API 開発）
- **勝ち筋**：API の仕様を見える化して共有
- **やらない**：内部メモを外に出しすぎる
- **頻度**：API がある時

### 4-3. AI・データ

#### Hugging Face（AI 拠点）
- **勝ち筋**：Space/Model/Dataset の"実物"を置く
- **やらない**：未整理の素材投げっぱなし
- **頻度**：AI 成果が出た時

#### Weights & Biases（実験記録）
- **勝ち筋**：実験ログが"研究として"残る
- **やらない**：毎回必ず記録（最初は余裕がある時）
- **頻度**：実験する時

#### Kaggle（戦闘力）
- **勝ち筋**：Notebook 共有で実力の証明
- **やらない**：コンペ沼に入る（目的を決める）
- **頻度**：月 1 で OK

#### Civitai（生成 AI 作品）
- **勝ち筋**：作品として見られる場所に置く
- **やらない**：権利が曖昧な素材の使用
- **頻度**：作品が出た時

### 4-4. 知識・コミュニティ

#### Stack Overflow / Teratail
- **勝ち筋**：回答で信用が貯まる
- **やらない**：炎上しそうな議論
- **頻度**：週 1 で 1 回答できれば十分

#### Reddit / Discord / Telegram
- **勝ち筋**：情報収集、海外コミュニティ接続
- **やらない**：追いかけすぎて制作時間を溶かす
- **頻度**：週 1 でまとめ読み

### 4-5. ドキュメント

#### Notion
- **勝ち筋**：ネタ帳、下書き、チェックリストの母艦
- **やらない**：Notion だけで完結（公開資産にしない）
- **頻度**：毎回

#### Docswell / Speaker Deck
- **勝ち筋**：スライドを"URL 資産"にする
- **やらない**：毎回スライドを作る
- **頻度**：話す/まとめる時

### 4-6. 発信・イベント

#### Zenn / Qiita
- **勝ち筋**：検索・共有に残る"長文の資産"
- **やらない**：毎回完璧を目指す、重複全文投稿で SEO を割る
- **頻度**：月 1 でも OK（続ける方が強い）

#### connpass
- **勝ち筋**：参加/登壇の履歴が信用になる
- **やらない**：イベント参加で満足して出さない
- **頻度**：イベントのたび

#### はてな / note
- **勝ち筋**：文化圏への接続、長文エッセイ/活動ログ
- **やらない**：技術記事の主戦場を増やしすぎる
- **頻度**：気が向いたら

### 4-7. キャリア・ビジネス

#### LAPRAS / Wantedly / LinkedIn
- **勝ち筋**：成果物 URL を束ねて"見える化"
- **やらない**：プロフィールだけ作って放置
- **頻度**：月 1 でリンク更新

#### PayPal
- **勝ち筋**：海外含む支払い導線を一本化（`paypal.me/fluxrozin`）
- **やらない**：最初から収益化に寄せすぎる
- **頻度**：必要時

### 4-8. SNS・クリエイティブ

#### X（速報）
- **勝ち筋**：制作過程/学びを小出しにする
- **やらない**：議論で消耗
- **頻度**：制作した日は 1 投稿

#### Bluesky / Threads
- **勝ち筋**：バックアップの配信面
- **やらない**：主戦場を増やしすぎる
- **頻度**：コピペで OK

#### Instagram
- **勝ち筋**：写真ポートフォリオ（Z9/Zf）
- **やらない**：技術発信の主戦場にする
- **頻度**：作品が撮れた時

#### TikTok
- **勝ち筋**：ショートで"作品"を見せる
- **やらない**：毎日投稿の義務化
- **頻度**：月 1 でも OK

#### YouTube
- **勝ち筋**：デモ・解説で信用が上がる
- **やらない**：撮影/編集に沼る（最初は画面収録で十分）
- **頻度**：四半期に 1 本でも価値がある

### 4-9. 競技・スキル

#### AtCoder
- **勝ち筋**：地力の証明
- **やらない**：制作を止めて競技だけやる
- **頻度**：週 1 の筋トレ枠

## 5. 再利用ルール（"1→5"の具体例）

| 中心（正本）      | 横展開（配信）                    | 補足                 |
| ----------------- | --------------------------------- | -------------------- |
| GitHub Repo       | Zenn 記事 → X 告知 → YouTube デモ | 記事は README の拡張 |
| Zenn 記事         | X スレッド / Bluesky / Threads    | 同じ要点を短く切る   |
| スライド          | Docswell + Speaker Deck           | 国内/海外で置き分け  |
| YouTube 動画      | TikTok ショート / X               | ショートは切り抜き   |
| HF/Civitai 成果物 | 記事 + 動画 + 作品一覧            | "作り方"が資産化     |

## 6. 段階的ロードマップ

最初から全部を完璧にしない前提で、成果が出やすい順に積み上げます。

### Phase 0: 防衛（1 日）

- **主要アカウントの MFA/復旧導線を固める**（優先：ドメイン/DNS、メール、GitHub、PayPal）
- **`fluxrozin.com` を「唯一の正しい入口（Single Entry Point）」として決める**

#### チェックリスト

- [ ] Cloudflare：MFA/復旧導線/DNSSEC（可能なら）を固める
- [ ] Microsoft：MFA/復旧導線を固め、`fluxrozin@outlook.com` を主要連絡先にする
- [ ] GitHub：2FA 必須化、復旧コード保管、公開メール/通知の整理
- [ ] PayPal：2 段階認証と `paypal.me/fluxrozin` の導線確認

### Phase 1: 統一（1 週間）

- **表示名 / アイコン / Bio / リンクを共通セットで統一する**
- **主要プロフィールのリンク構造を整える**（各サービス → `fluxrozin.com`）

#### チェックリスト

- [ ] アイコン/ヘッダー/配色の"ブランドキット"を決める（Gravatar も更新）
- [ ] Bio（短文/中文）を作り、全サービスへ展開する
- [ ] 主要プロフィールのリンクを `fluxrozin.com` に統一する
- [ ] `docs/01_services.md` の「NEW」サービスを上から順にプロフィール整備する

### Phase 2: 配信（2〜4 週間）

- **長文（Zenn/Qiita）→ 短文（X/Threads）→ 動画（YouTube/TikTok）の再利用パイプラインを作る**
- **AI 成果物（HF/Civitai）を記事・動画へ接続し、実績として蓄積する**

#### チェックリスト

- [ ] Zenn/Qiita：自己紹介 + 代表記事の土台を作る
- [ ] Speaker Deck/Docswell：代表スライドの置き場を用意する
- [ ] YouTube/TikTok：最初の 1 本（自己紹介/代表作）を出す
- [ ] Hugging Face/Civitai：成果物を 1 つ公開し、記事/動画へリンクする

### Phase 3: 蓄積（継続）

- **GitHub / スライド / 記事 / 登壇履歴を相互リンクし、検索可能な資産にする**
- **週次レビューで「やること」を減らし、「残ること」を増やす**

#### チェックリスト

- [ ] connpass：参加/登壇履歴を整備し、スライド/記事に接続する
- [ ] LAPRAS/Wantedly/LinkedIn：職務経歴と成果物のリンクを最新化する
- [ ] 週次レビュー（`templates/weekly-review.md`）を回す

## 7. 週次・月次の運用ルーティン

### 7-1. 週 1 回（30 分）：最小のアウトプット

```
□ 今週学んだことを 1 つ、GitHub に Markdown で残す
□ 余裕があれば Qiita/Zenn に清書
□ X で「今週やったこと」を 1 投稿
```

### 7-2. 月 1 回（1 時間）：整理と更新

```
□ fluxrozin.com/works を更新
□ GitHub の README を整理
□ LAPRAS/LinkedIn/Wantedly にリンク追加
□ 来月やりたいことを 1 つ Notion に書く
```

### 7-3. 断続になった時の再起動（15 分）

```
□ Notion のネタ帳を開く
□ 一番気になっていることを 1 つ選ぶ
□ GitHub に Issue を 1 つ作る（「◯◯を調べる」で OK）
□ 30 分だけ作業する
□ 何か 1 つ公開する（X に「やる宣言」だけでも可）
```
